## Session 2: Security

### [CRITICAL] PGlite quiescing before backup copy

- **What:** Backup service must stop active PGlite connections or call checkpoint before copying data directory.
- **Why:** Copying a live PGlite data directory while writes are in-flight can produce a corrupt backup. File-level copy is not atomic.
- **Pros:** Prevents silent data corruption in backups — the entire point of the backup feature.
- **Cons:** May add 1-2 seconds of write unavailability during backup window.
- **Context:** PGlite stores data as files in `data/pglite/`. The backup service copies this directory, encrypts it with AES-256, and writes to `data/backups/`. Without quiescing, a backup taken during an active INSERT/UPDATE could capture a half-written WAL file. Research whether PGlite exposes a `CHECKPOINT` command or whether closing and reopening the connection pool is sufficient.
- **Depends on:** Backup service implementation (Session 2).
